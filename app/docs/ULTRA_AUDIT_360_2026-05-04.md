### 🔴 Finding #F1 — Numéros d'urgence médicaux Tunisie/Algérie/Maroc possiblement faux

- **Sévérité** : **CRITICAL** (risque vital, app médicale)
- **Axe** : 1 (sécurité fonctionnelle/clinique)
- **Fichier** : `app/src/utils/clinicalChecks.ts:225-249, 275-283`
- **Description** : Le module `clinicalChecks` produit des `ClinicalAlert` de niveau `critical` pour 4 cas pouvant être engageants : hypoglycémie sévère (<3.5 mmol/L), hyperglycémie sévère (>10 mmol/L), hypertension sévère (sys≥160 ou dia≥110, drapeau pré-éclampsie), et autres. Pour ces cas, `HealthDashboardScreen.tsx:202-211` affiche un bouton "Appeler les urgences (NUMÉRO)" qui exécute `Linking.openURL('tel:' + number)`. La table `EMERGENCY_NUMBERS` mappe :
  - **TN/TUNISIA/TUNISIE → '197'** : 197 est le numéro de la **Sûreté Nationale (police)** en Tunisie. Le SAMU médical tunisien est **190**, la Protection Civile (ambulances) est **198**. Le commentaire ligne 275 affirme "Tunisia SAMU" — c'est inexact.
  - **DZ/ALGERIA/ALGERIE → '14'** : à vérifier ; en Algérie le SAMU est **14** *ou* **115** selon les régions (il n'y a pas de numéro national unique fiable pour le médical urgent jusqu'à fin 2024).
  - **MA/MOROCCO/MAROC → '141'** : à vérifier ; au Maroc l'**SAMU** est **141** dans certaines wilayas mais le numéro générique d'ambulance est **150** (Protection Civile).
  - **FR/FRANCE → '15'** : ✅ correct (SAMU national France).
- **Extrait** :
  ```ts
  // clinicalChecks.ts:225-235
  export const EMERGENCY_NUMBERS: Readonly<Record<string, string>> = Object.freeze({
      FR: '15',
      FRANCE: '15',
      TN: '197',          // ⚠️ POLICE, pas SAMU. SAMU TN = 190
      TUNISIA: '197',     // ⚠️
      TUNISIE: '197',     // ⚠️
      MA: '141',          // ⚠️ à vérifier (PC = 150)
      ...
      DZ: '14',           // ⚠️ à vérifier
  ```
  ```ts
  // clinicalChecks.ts:275-283
  /**
   *   ar | tn → 197 (Tunisia SAMU — current default for the AR/TN UI)  ⚠️ 197 ≠ SAMU
   */
  const LOCALE_FALLBACK: Readonly<Record<string, string>> = Object.freeze({
      ar: '197',
      tn: '197',
      fr: '15',
  });
  ```
- **Impact** : pendant une crise hypertensive (≥160/110), une utilisatrice tunisienne tape "Appeler les urgences" → la police décroche, doit transférer ou rappeler une ambulance → délai de plusieurs minutes. **Pour une pré-éclampsie, la fenêtre thérapeutique est critique**. L'app de santé est censée *réduire* le délai, pas l'augmenter.
- **Recommandation** :
  1. **Bloquer la release publique** pour tout pays autre que la France tant que le code n'est pas vérifié par un médecin local et/ou les sites officiels (Ministère de la Santé TN, MA, DZ).
  2. Court terme : remplacer TN par '190' (SAMU) ou '198' (Protection Civile / ambulance) et faire valider par 1 médecin tunisien.
  3. Sécurité par défaut : si confiance < élevée → renvoyer `null` et masquer le bouton (le commentaire FR du code l'admet déjà : « we never make up a number »). Appliquer ce principe strictement.
  4. Ajouter dans le test suite (`clinicalChecks.test.ts`) une fixture confirmée par un médecin documentée avec le lien officiel.
- **Cycles de healing** : 0 (à traiter dans cycle dédié médical).

### 🔴 Finding #F2 — Manifeste Privacy iOS gravement incomplet

- **Sévérité** : **CRITICAL** (rejet App Store probable + non-conformité GDPR Apple)
- **Axe** : 7 (store readiness) + 1 (GDPR)
- **Fichier** : `app/app.json:28-41`
- **Description** : `app.json` ne déclare qu'**un seul** type de données collectées (`NSPrivacyCollectedDataTypeEmailAddress`). Or l'application collecte et stocke côté serveur (Firestore) AU MINIMUM :
  - Identifiants (Firebase UID, propagé en breadcrumbs Sentry → `logger.ts:228-238`).
  - **Données de santé** : poids (`healthMetrics`, `weight_entries`), tension (`healthMetrics` type=`blood_pressure`), glycémie (`glucoseMetrics`), symptômes (`symptomsLog`, type sensible : `nausea`, `swelling`, `breathlessness`).
  - **Reproductive Health** (Apple `NSPrivacyCollectedDataTypeHealthAndFitness` + `SensitiveInfo`) : DDR, date de conception, ovulation, fenêtre fertile (`UserProfile.ovulationDate`, `fertileWindowStart`, `fertileWindowEnd`), statut TTC.
  - User Content : `notes` saisies sur poids/TA/glucose/symptômes ; tâches custom ; rendez-vous médicaux.
  - Usage Data : événements analytics (no-op au runtime mais déclarés dans le code et le binaire).
  - Diagnostics : Sentry crash reports avec UID + breadcrumbs + extras.
  - Location : non géoloc précise mais **pays + ville** stockés (`UserProfile.country`, `city`) → catégorie "Coarse Location" pour Apple.
  - Contact Info Other : `emergencyContacts` (numéros téléphone tiers).
- **Extrait** :
  ```json
  // app.json:28-40
  "privacyManifests": {
      "NSPrivacyCollectedDataTypes": [
          {
              "NSPrivacyCollectedDataType": "NSPrivacyCollectedDataTypeEmailAddress",
              ...
          }
      ],
      "NSPrivacyAccessedAPITypes": []
  }
  ```
- **Impact** : (a) rejet probable de la review App Store (Privacy Manifest a été rendu obligatoire pour SDKs en 2024 et étendu progressivement aux apps depuis 2025, l'incohérence avec App Privacy form Apple est repérée par les reviewers) ; (b) en cas d'audit CNIL ou DPO, c'est une faille manifeste de conformité ; (c) si l'app est listée sur l'App Store sans déclaration correcte de "Health & Fitness", elle sort des apps santé recommandées (perte de visibilité + crédibilité).
- **Recommandation** :
  1. Compléter `NSPrivacyCollectedDataTypes` avec : `Name`, `EmailAddress`, `PreciseLocation`/`CoarseLocation` (pays/ville), `HealthAndFitness`, `SensitiveInfo`, `OtherUserContent`, `UserID`, `DeviceID`, `CrashData`, `OtherDiagnosticData`, `ProductInteraction`.
  2. Pour chaque type : `NSPrivacyCollectedDataTypeLinked: true` (lié à l'utilisateur via Firebase), `NSPrivacyCollectedDataTypeTracking: false`, et `Purposes` : `AppFunctionality`.
  3. Compléter `NSPrivacyAccessedAPITypes` : `NSPrivacyAccessedAPICategoryUserDefaults` (AsyncStorage), `NSPrivacyAccessedAPICategoryDiskSpace` (expo-sqlite), `NSPrivacyAccessedAPICategoryFileTimestamp` si `expo-print` est utilisé.
  4. Aligner App Privacy form sur App Store Connect.
- **Cycles de healing** : 1 cycle suffit (déclaratif, sans risque code).

### 🔴 Finding #F3 — Pas de Firebase App Check (DoS/abus quota possible)

- **Sévérité** : **CRITICAL** (sécurité plateforme + coût exploitation)
- **Axe** : 1 (sécurité backend)
- **Fichier** : `app/src/config/firebase.ts:1-42` (absence de `initializeAppCheck`), `firestore.rules:126-136`, `eas.json:31-36`, `.env`
- **Description** : la clé `EXPO_PUBLIC_FIREBASE_API_KEY` est dans le bundle (normal pour Web SDK). Sans App Check, n'importe qui peut faire :
  ```bash
  curl https://firestore.googleapis.com/v1/projects/pregnancy-app-1f939/databases/(default)/documents/articles \
       -H "X-Goog-Api-Key: AIzaSyBC7HcOFJqICM1JOXUa8pUtdCBF0IUm4aU"
  ```
  et :
  - Lire intégralement les 11 collections statiques (`articles`, `articlesAntigravity`, `tips`, `weeks`, `babyMessages`, `supplements`, `redFlags`, `chatbotSuggestionsAG`, `weeklyTasks`, `reminderTemplates`, `calendarTemplates`) — chacune `allow read: if true` (`firestore.rules:126-136`). C'est **du contenu propriétaire** (40 semaines × ~10 suggestions × 4 langues + DATA_PACK V3) qui peut être scrapé.
  - Pingoler `signInWithEmailAndPassword` à grande échelle (le rate-limiter client `authLimiter` de `rateLimiter.ts:132-135` est purement local).
  - Drainer le quota Firestore gratuit (50k reads/jour par défaut) en quelques minutes → l'app devient inutilisable en production pour les vraies utilisatrices.
- **Extrait** :
  ```ts
  // firebase.ts:17 — pas de initializeAppCheck
  const app = initializeApp(firebaseConfig);
  const auth = initializeAuth(app, { persistence: getReactNativePersistence(...) });
  export const db = initializeFirestore(app, { localCache: persistentLocalCache() });
  // ❌ Aucun import de firebase/app-check, aucun ReCAPTCHAEnterpriseProvider, etc.
  ```
  ```js
  // firestore.rules:126-136 — 11 lectures publiques sans auth
  match /articles/{docId} { allow read: if true; allow write: if false; }
  match /weeks/{docId}    { allow read: if true; allow write: if false; }
  ...
  ```
- **Impact** :
  1. **Vol de contenu** : les 1.46 MB d'`articles_db.json`, weeks, supplements, etc. sont récupérables en quelques requêtes par un concurrent ou un scraper.
  2. **DoS quota** : un attaquant motivé peut épuiser le quota gratuit ou faire flamber la facture Firestore en mode pay-as-you-go.
  3. **Brute force auth** : l'`authLimiter` client est trivialement contournable.
- **Recommandation** :
  1. Activer **Firebase App Check** : `initializeAppCheck(app, { provider: new ReactNativeFirebaseAppCheckProvider({ apple: { provider: 'appAttestWithDeviceCheckFallback' }, android: { provider: 'playIntegrity' } }) })`. Note : nécessite `@react-native-firebase/app-check` (pas le Web SDK) → migration partielle vers le SDK natif Firebase.
  2. Resserrer les rules pour `static content` :
     ```js
     match /articles/{docId} { allow read: if isAuthenticated(); ... }
     ```
     ou au minimum ajouter `&& request.auth != null` après vérif que l'écran de découverte ne tourne plus en pre-auth.
  3. Migrer le rate-limit auth/écriture vers une **Cloud Function** ou activer Firebase Auth's "Email enumeration protection" + "App Check" enforcement.
  4. À défaut d'App Check : configurer **API key restrictions** dans GCP Console (allowlist par bundle ID iOS / SHA-1 Android).
- **Cycles de healing** : ≥ 2 (chantier non trivial — App Check natif demande prebuild + tests réels TestFlight/Internal Testing).

### 🟠 Finding #F4 — `deleteAccount` ne gère pas `auth/requires-recent-login` (suppression échoue silencieusement)

- **Sévérité** : **HIGH** (GDPR — droit à l'effacement bloqué)
- **Axe** : 1 (auth) + 3 (data lifecycle)
- **Fichier** : `app/src/context/AuthContext.tsx:513-619`
- **Description** : Firebase exige une session récente (généralement < 5 min) pour `firebaseUser.delete()`. Au-delà → throw `auth/requires-recent-login`. Le code purge d'abord les collections Firestore (lignes 539-589) puis tente `firebaseUser.delete()` (ligne 592). Si l'auth fail, **le compte Firebase Auth survit mais toutes les données Firestore ont été détruites** → l'utilisateur peut se reconnecter, voir un compte vide, mais l'UID reste alloué côté Auth.
- **Extrait** :
  ```ts
  // AuthContext.tsx:534-593
  // 1. Delete Firestore Data (User Profile) - Do this first or alongside collections
  const userDocRef = doc(db, 'userProfiles', uid);
  await deleteDoc(userDocRef);
  ...
  for (const colName of topLevelCollectionsToDelete) { /* batch delete */ }
  ...
  // 2. Delete Firebase Auth User (Must happen LAST so Firestore rules still apply)
  await firebaseUser.delete();   // ❌ peut throw 'auth/requires-recent-login'
  ```
  Aucun `try/catch` autour de `firebaseUser.delete()`, aucun appel à `reauthenticateWithCredential` préalable. Le `try` global (615-619) rethrow → l'utilisateur voit une alerte d'erreur générique sans comprendre pourquoi.
- **Impact** : violation potentielle de RGPD Art. 17 (droit à l'effacement) — les données sont effacées mais l'identifiant Auth subsiste, et si la prochaine tentative est encore en `requires-recent-login`, l'utilisateur ne peut pas finaliser sa suppression.
- **Recommandation** :
  1. Inverser l'ordre : 1) tenter `firebaseUser.delete()`, 2) si OK alors purger Firestore, 3) si KO → afficher un dialogue "Veuillez vous reconnecter pour confirmer la suppression" + flow `reauthenticateWithCredential(firebaseUser, EmailAuthProvider.credential(email, password))`.
  2. Documenter dans `PrivacyPolicyScreen` que le délai d'effacement est immédiat *sous condition de session récente*.
  3. Ajouter test (`AuthContext.gdpr.test.ts`) : mock `firebaseUser.delete()` pour throw `auth/requires-recent-login` → vérifier que Firestore n'est pas purgé prématurément.
- **Cycles de healing** : 1 cycle.

### 🟠 Finding #F5 — Permissions Android dangereuses et incohérentes avec `app.json`

- **Sévérité** : **HIGH** (rejet Play Store probable)
- **Axe** : 7 (store readiness) + 1 (sécurité)
- **Fichier** : `app/android/app/src/main/AndroidManifest.xml:2-9`, `app/app.json:50-55`
- **Description** :
  - **`SYSTEM_ALERT_WINDOW`** déclaré dans `AndroidManifest.xml:7` mais **PAS** dans `app.json` android.permissions ni utilisé dans aucun `import` du code (`grep -rn "SYSTEM_ALERT" src/` → 0 hit). C'est l'une des permissions les plus surveillées par Google Play (utilisée par malware/clickjacking) ; déclarer sans usage entraîne un rejet ou un avertissement.
  - **`READ_EXTERNAL_STORAGE`** + **`WRITE_EXTERNAL_STORAGE`** déclarées (lignes 4, 9) — `WRITE_EXTERNAL_STORAGE` est dépréciée depuis Android 10 (API 29) et **ignorée à partir d'Android 11** (target SDK 30+). Sa présence sans `maxSdkVersion="29"` est un rouge sur la review.
  - **`SCHEDULE_EXACT_ALARM`** déclarée — depuis Android 14 (API 34) requiert une justification pour les apps "alarme/calendrier" via le formulaire Play Store. Une app de rappels grossesse peut le justifier mais nécessite une demande explicite.
  - Incohérence : `app.json:50-55` ne liste que `RECEIVE_BOOT_COMPLETED`, `SCHEDULE_EXACT_ALARM`, `VIBRATE`, `POST_NOTIFICATIONS`. Les 5 autres permissions du manifest natif viennent de prebuild + plugins. Soit le manifest est édité à la main, soit le prebuild contient des résidus.
- **Extrait** :
  ```xml
  <!-- AndroidManifest.xml:2-9 -->
  <uses-permission android:name="android.permission.INTERNET"/>
  <uses-permission android:name="android.permission.POST_NOTIFICATIONS"/>
  <uses-permission android:name="android.permission.READ_EXTERNAL_STORAGE"/>
  <uses-permission android:name="android.permission.RECEIVE_BOOT_COMPLETED"/>
  <uses-permission android:name="android.permission.SCHEDULE_EXACT_ALARM"/>
  <uses-permission android:name="android.permission.SYSTEM_ALERT_WINDOW"/>  <!-- ⚠️ -->
  <uses-permission android:name="android.permission.VIBRATE"/>
  <uses-permission android:name="android.permission.WRITE_EXTERNAL_STORAGE"/>  <!-- ⚠️ -->
  ```
- **Impact** : avertissement / rejet Play Store + Data Safety form invalidée + alerte AVIS utilisateur ("Cette app peut s'afficher au-dessus des autres applications").
- **Recommandation** :
  1. Régénérer le manifest depuis `app.json` (`npx expo prebuild --clean`) après avoir verrouillé la liste exacte des permissions Android.
  2. Ajouter dans `app.json`:
     ```json
     "android": {
       "blockedPermissions": ["android.permission.SYSTEM_ALERT_WINDOW",
                              "android.permission.WRITE_EXTERNAL_STORAGE",
                              "android.permission.READ_EXTERNAL_STORAGE"]
     }
     ```
  3. Pour `SCHEDULE_EXACT_ALARM` : préparer la justification "Alarms & Reminders" pour Play Console + envisager `USE_EXACT_ALARM` (Android 13+) qui ne demande pas la permission utilisateur si l'usage est réellement calendrier.
- **Cycles de healing** : 1 cycle (config + prebuild).

### 🟠 Finding #F6 — Export GDPR via `Share.share` envoie le JSON brut PII

- **Sévérité** : **HIGH** (fuite PII via canal tiers)
- **Axe** : 1 (sécurité) + 3 (GDPR)
- **Fichier** : `app/src/services/dataExportService.ts:182-201`
- **Description** : `exportUserData` utilise `Share.share({ message: jsonString })`. Le JSON contient TOUTES les données santé (poids, TA, glycémie, symptômes), pré-chargé dans le presse-papier / paquet de partage natif. Sur iOS, ce share peut être :
  - copié dans le presse-papier d'apps tierces sans consentement explicite,
  - envoyé via AirDrop sans chiffrement (en clair),
  - sauvegardé par des intégrations clipboard (Keyboard tiers, gestionnaires de mots de passe).
  Sur Android, idem avec n'importe quelle target d'Intent.
- **Extrait** :
  ```ts
  // dataExportService.ts:182-201
  export const exportUserData = async (user: any, profile: any, t: any) => {
      const data = await generateExportData(user, profile);
      const jsonString = JSON.stringify(data, null, 2);
      const result = await Share.share({
          message: jsonString,                                           // ❌ PII brut
          title: `mama-bebe-export-${new Date().toISOString().split('T')[0]}.json`
      });
  ```
- **Impact** : données médicales sensibles (symptômes, glycémie, TA, ovulation) potentiellement piégées par des clipboard managers / claviers tiers / partagées par erreur dans une mauvaise conversation. RGPD Art. 32 demande un niveau de sécurité approprié au risque — cette interface ne le fournit pas.
- **Recommandation** :
  1. Préférer un export *fichier* via `expo-print` (PDF) ou `expo-file-system` (JSON sur disk) + `expo-sharing.shareAsync(uri, { mimeType: 'application/json', UTI: 'public.json' })` — ne passe pas par le presse-papier, l'utilisateur choisit explicitement la destination.
  2. Optionnel : chiffrer l'export avec une passphrase utilisateur (AES via `react-native-quick-crypto` ou `expo-crypto`).
  3. Afficher un dialogue d'avertissement avant export : "Ce fichier contient vos données de santé. Choisissez une destination sécurisée."
- **Cycles de healing** : 1 cycle.

### 🟠 Finding #F8 — Firestore rules sans validation de champs (deny-by-default partiel)

- **Sévérité** : **HIGH** (intégrité des données)
- **Axe** : 1 (sécurité backend)
- **Fichier** : `app/firestore.rules` (intégralité)
- **Description** : les règles sont en mode "implicit deny" (ce qui est OK), MAIS aucune `match` ne valide :
  - le **schema** des documents (`request.resource.data.foo is string`) — un attaquant ou un bug client peut écrire `weight: { $ref: 'something' }` dans `weight_entries`.
  - la **taille** des documents (limite Firestore = 1 MB par doc, mais on peut imposer plus restrictif).
  - les **timestamps** (`request.time`) pour empêcher l'antédatation des entrées.
  - les **plages numériques** : un poids saisi à 99999 kg passe les rules (les checks sont uniquement client-side dans `validation.ts:118-143`).
- **Extrait** :
  ```js
  // firestore.rules:65-70 — aucune validation de schema
  match /healthMetrics/{metricId} {
      allow read: if isOwnerOfResource();
      allow create: if isOwnerOfData();
      allow update: if isOwnerOfResource() && isOwnerOfData();
      allow delete: if isOwnerOfResource();
  }
  ```
  Comparer à une règle robuste :
  ```js
  allow create: if isOwnerOfData()
      && request.resource.data.type in ['weight','blood_pressure']
      && request.resource.data.date is string
      && request.resource.data.week is number
      && request.resource.data.week >= 0 && request.resource.data.week <= 45
      && (request.resource.data.notes == null || request.resource.data.notes.size() <= 500);
  ```
- **Impact** : (a) un client compromis peut polluer le dataset utilisateur ; (b) un autre utilisateur (qui maîtrise l'API) peut consommer beaucoup de stockage en injectant des notes longues ; (c) cohérence des graphes santé compromise si un poids fantaisiste est inséré.
- **Recommandation** : ajouter validation par collection (au minimum `type`, `date`, `week`, taille de `notes`) + 1 cycle de tests `firestore-rules-test`. Le repo a déjà `firestore.rules` versionné mais aucun test associé (vérifier dans `__tests__` — pas trouvé).
- **Cycles de healing** : 1-2 cycles (chantier rules + suite de tests `@firebase/rules-unit-testing`).
